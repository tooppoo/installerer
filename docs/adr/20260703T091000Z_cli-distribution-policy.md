# CLI Distribution Policy

- Status: Accepted
- Created: 2026-07-03T09:10:00Z

## Context

`installerer` started as a browser-based installer generator. The CLI version needs a distribution model that preserves the boundary between the generator and the generated installer runtime.

The CLI itself is a developer-facing generator tool. The generated `install.sh` remains the end-user installer artifact for target binaries.

This ADR records the CLI distribution decision from [Issue #29](https://github.com/tooppoo/installerer/issues/29).

## Decision

`installerer` CLI is distributed primarily as a Bun-compiled standalone executable.

The canonical executable is the standalone binary produced by Bun compile. The canonical install artifact consumed by the official installer is a release archive that contains that standalone executable.

The official installer for `installerer` is generated by `installerer` itself. This is dogfooding in the limited sense that the official installer is generated ahead of time by `installerer`; users do not generate the installer during installation.

The npm package is an auxiliary JavaScript ecosystem distribution. It is a Node.js CLI package, not a GitHub Releases binary downloader, and not the canonical binary distribution.

The CLI remains generator-only in v0. Supported commands are limited to `init`, `validate`, `generate`, `doctor`, `--version`, and `--help`. Commands such as `installerer install`, `installerer run`, and `installerer exec` are explicitly out of scope.

## Implementation Split

Implementation is split across dedicated issues:

- [Issue #81](https://github.com/tooppoo/installerer/issues/81): npm Node.js CLI distribution
- [Issue #82](https://github.com/tooppoo/installerer/issues/82): Bun standalone executable and release archive creation
- [Issue #80](https://github.com/tooppoo/installerer/issues/80): target variable support
- [Issue #83](https://github.com/tooppoo/installerer/issues/83): official installer implementation

Prerequisite order:

- Issue #80 starts after Issue #76.
- Issue #83 starts after Issue #76, Issue #80, and Issue #82.
- Issue #81 is not a hard blocker for Issue #76 or Issue #80.
- Issue #82 is not a hard blocker for Issue #76 or Issue #80.

## Release Archive Naming

`installerer` release archive names do not use Bun compile target labels.

Bun compile targets are build-job-internal labels. Public release archive names use the OS / architecture labels expected by the generated installer.

v0 official installer target archive names are:

- `installerer_{version}_Linux_x86_64.tar.gz`
- `installerer_{version}_Linux_arm64.tar.gz`
- `installerer_{version}_Darwin_x86_64.tar.gz`
- `installerer_{version}_Darwin_arm64.tar.gz`

Each archive contains the CLI executable at the path `installerer`.

Raw standalone executable assets may be attached to GitHub Releases later, but they are not required by this decision.

The release job maps Bun compile targets to public archive labels:

| Bun compile target | Public archive label |
|---|---|
| `bun-linux-x64-baseline` | `Linux_x86_64` |
| `bun-linux-arm64` | `Linux_arm64` |
| `bun-darwin-x64` | `Darwin_x86_64` |
| `bun-darwin-arm64` | `Darwin_arm64` |

## Official Installer

The official installer implementation is handled separately in Issue #83.

It depends on:

- Issue #76, for architecture label mapping and validation
- Issue #80, for target variable support
- Issue #82, for release archive creation, checksum file generation, and version file generation or placement

The official installer uses an `installerer.self.json`-style config and generates an `install.sh`. CI checks that the generated `install.sh` remains in sync with its source config.

The generated installer verifies the release archive checksum before extraction and binary placement. This checksum verification checks artifact integrity against the fetched checksum file, but it does not prove publisher identity or supply-chain provenance.

## npm Distribution

The npm package uses `@philomagi/installerer`.

The npm package is a Node.js CLI package and must not download GitHub Releases binaries as part of normal operation.

For npm publishing, `private: true` may be removed from the npm-published package configuration.

Node.js CLI code must avoid Bun runtime APIs and keep CLI-specific file IO, stdout, stderr, and exit-code behavior outside the runtime-independent generator core.

## Alternatives Considered

### npm As The Canonical Distribution

This would fit JavaScript tooling, but it would make Node.js part of the primary CLI runtime requirement. That is not selected.

### npm Package As A Binary Downloader

The npm package could download the canonical GitHub Releases binary during install or first execution. This is not selected for v0 because it adds supply-chain, caching, offline-use, and install-script complexity.

### Raw Executable As The Official Installer Artifact

The official installer could fetch raw executables directly. This is not selected because the generated installer runtime already works with release archives, checksum lookup, extraction, and binary placement.

### Bun Target Labels As Public Archive Names

Public release archive names could use Bun compile target labels directly. This is not selected because it would expose build-tool-specific labels as part of the public artifact naming contract.

### CLI As A Package Manager

The CLI could implement direct package installation, upgrade, or uninstall behavior. This is not selected. `installerer` remains an installer generator.

## Consequences

### Positive Consequences

- The primary CLI distribution does not require Node.js or Bun at runtime.
- The npm package remains simple and explicit as a JavaScript ecosystem integration.
- The official installer dogfoods generated installers without requiring installer generation at install time.
- Public release archive names stay aligned with generated installer OS / architecture labels.
- CLI responsibility remains limited to generator behavior.

### Negative Consequences

- The project must maintain both standalone executable distribution and Node.js npm distribution.
- Binary and npm CLI behavior must be tested for consistency.
- The release job must map Bun compile targets to public archive labels.
- Official installer work must wait for target variable support and release archive generation.

### Neutral Consequences

- Dev Container Feature distribution is deferred.
- Windows native installer support is out of scope for v0.
- Signature and provenance verification remain future work.
